4 · 推荐执行顺序与理由

第一步: M1a (3-5 天) — 截图驱动的最薄切片
  • S2/S4 直接给出 38 hooks 的视觉真值,逐字段截图比对即可验收
  • 不依赖 modules.fee/ea/dem,不依赖 parallelmodelmanager,不依赖外挂 EXE
  • 每补一个 hook 撞到 EcuUtilsBase 缺什么就补什么(demand-driven 自然填 L1)
  • 命中 memory 纪律:99% 复刻 + 整文件 paraphrase + plan 先写 + deploy 前 unit test
第二步: M1a' (0.5 天) — MemIf 同模式扫尾
  • 3 个 hook,模板已立,机械填空
  • 顺手把 MemIfValidator/Generator stub 完成度推到 M1 范围
第三步: M1b (1.5-2 周) — 截图 S1 阻塞解锁
  • 建 modules.fee + modules.ea bundle(各参考 ~80-200 类,但 NvM 只用其 ECUC schema + 容器列举,不需要全实装)
  • L3 pal.librarymanager + L4 bswbuilder.librarymanager 配套
  • 10 ReferenceValueScopeUIDefinition 全实装
  • 验收: S1 dropdown 工作 + 8 demevent ref 工作(若 dem 同步建则一并)
不推荐的反例:
  • ❌ 先做 M4 RTE 联动: parallelmodelmanager 整 bundle ~50 类是 L4 巨石, vertical slice 切不动, 一次性投入 2 周看不到截图绿
  • ❌ 先做 M3 generator: byte-equal 真值已在 Phase 1.5 采到, 但 NvMGenerator 连外挂 EXE 桥都没建, 先做意义低
  • ❌ 纯自下而上补 L1-L4: 几万行框架代码无截图驱动验证, 过度设计 + 接口对不上 = 必返工

5 · 关键判断(收口)

截图分析后, 推荐从 M1 拆为 M1a + M1b 两片, 先做 M1a:
  1. 3-5 天可见绿色截图(对照 S2/S4)
  2. 不引入新 bundle,所有改动在 modules.nvm + mal/mal.modelutils 内
  3. 暴露 L1 mal.modelutils 真实需求,demand-driven 自然填(不预补 32 方法整集)
  4. 下一片 M1b 才碰跨模块 bundle 新建(modules.fee/ea + librarymanager 双层)
  5. M2/M3/M4 不在当前推荐范围,完成 M1a 后再选下一片
