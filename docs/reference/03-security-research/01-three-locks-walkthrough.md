## 一、核心心智模型：**三套互不相关的加密**

这是最关键的一点 —— 之前你可能把它们当成一件事，其实是**三个独立的锁**，各保护各的：

```
┌──────────────────────────────────────────────────────┐
│  🔐 锁一：BitAnswer 软锁                              │
│     保护的是：整个 exe 能不能启动                     │
│     技术：机器指纹绑定 + 授权码                       │
├──────────────────────────────────────────────────────┤
│  🔐 锁二：.class 字节码加密（XOR 0x21）              │
│     保护的是：iSoft 的 Java 代码别被看懂              │
│     技术：每个 .class 文件每字节 ^ 0x21              │
├──────────────────────────────────────────────────────┤
│  🔐 锁三：PuHuaLicense.lic 授权文件                  │
│     保护的是：你能用哪几颗 MCU 芯片                   │
│     技术：XML 正文 + AES 加密的自签名                 │
└──────────────────────────────────────────────────────┘

另外还有个独立的（跟启动流程无关）：
┌──────────────────────────────────────────────────────┐
│  🔐 锁四：ARXML 工程文件完整性                        │
│     保护的是：你别手改工程文件绕过 GUI                │
│     技术：SHA-256 哈希 + AES 加密哈希表               │
└──────────────────────────────────────────────────────┘
```

## 二、每一把锁的钥匙在哪儿

| 锁 | 加密内容 | 解密钥匙在哪 | 谁用钥匙 |
|---|---|---|---|
| 🔐 锁一 BitAnswer | 没加密东西，只检查"你是不是授权机器" | BitAnswer 服务器（网上）+ 你电脑里一个数字证书文件 | BitAnswer DLL（`00018990_*.dll`） |
| 🔐 锁二 XOR | iSoft 所有 `.class` 文件 | 固定常数 `0x21`（硬编码在 DLL 里） | BitAnswer DLL（顺便兼职） |
| 🔐 锁三 PuHuaLicense | 你的授权矩阵（能用哪几颗 MCU） | AES 密钥 `secret.key`（装在软件里） | iSoft 的 Java 代码 |
| 🔐 锁四 ARXML | 你的工程文件哈希 | AES 密钥 `"SHDWARJHSCYREKFE"`（硬编码在代码里） | iSoft 的 Java 代码 |

### 重点观察

- **锁一**：钥匙在服务器（你不能造）
- **锁二/三/四**：钥匙都**跟软件装在一起**，所以从技术上"保密"不了 —— 逆向一下就拿出来了

## 三、启动时间线：从双击 exe 到主界面出现

```
⏱ 时刻 T0：你双击 ORIENTAIS.exe
════════════════════════════════════════════════════════

T1 ──► ① Windows 启动 exe，读取 .ini 配置
        .ini 里写着 "-agentlib:decrypt"
        意思："启动 Java 时先加载 BitAnswer DLL"

T2 ──► ② JVM（Java 虚拟机）启动，加载 BitAnswer DLL

        ┌─────────────────────────────────────┐
        │ 🔐 锁一 BitAnswer 开始判断         │
        │                                     │
        │ 本地有没有"数字证书"？               │
        │   有 → 验证签名 + 机器指纹对得上 → ✓ │
        │   没 → 弹登录窗口，要你输授权码 → 联网│
        │        → 服务器下发证书 → 写本地 → ✓ │
        └─────────────────────────────────────┘

T3 ──► ③ JVM 开始加载 Java 类（.class 文件）

        JVM 读出来一看：
        "魔数是 EB DF 9B 9F？不对啊，正常是 CA FE BA BE"

        ┌─────────────────────────────────────┐
        │ 🔐 锁二 XOR 解密                     │
        │                                     │
        │ DLL 把每个字节 ^ 0x21：             │
        │   EB ^ 21 = CA                      │
        │   DF ^ 21 = FE                      │
        │   9B ^ 21 = BA                      │
        │   9F ^ 21 = BE                      │
        │                                     │
        │ 交还给 JVM，JVM："哦这就对了"       │
        └─────────────────────────────────────┘

        ★ 每加载一个 class 都走一遍这步
        ★ 加载了 4809 个 class → 解了 4809 次

T4 ──► ④ iSoft 的 Java 代码开始跑
        Eclipse RCP 框架初始化
        主界面开始画

T5 ──► ⑤ 用户点需要 MCU 列表的地方
        iSoft 的 AutocoreCoordinator.getMCUs() 被调用

        ┌─────────────────────────────────────┐
        │ 🔐 锁三 PuHuaLicense 校验            │
        │                                     │
        │ 读 configuration/PuHuaLicense.lic   │
        │ 读 configuration/secret.key         │
        │                                     │
        │ 用 secret.key 对 .lic 最后一行的     │
        │ AES 加密串解密                       │
        │                                     │
        │ 解密结果 == .lic 的 XML 正文？       │
        │   相等 → 文件没被改 → ✓             │
        │                                     │
        │ 读 .lic 里的 3 个 UUID →             │
        │ 翻译成 S32K148 / U2A16 / TC397      │
        └─────────────────────────────────────┘

T6 ──► ⑥ 主界面 MCU 下拉框显示 3 颗芯片，你开始干活

        （★ 锁四 ARXML 只有你保存/打开工程时才启动，
           跟启动流程无关。）
```

## 四、锁一和锁二有直接关系吗？

**表面看有关系，实际上没有。** 分三个维度讲。

### 维度 1：物理打包 —— 是的，打在一起

两把锁的逻辑**都在同一个 DLL 里**（`00018990_00000004_x64.dll`），都由同一个 JVM 启动参数 `-agentlib:decrypt` 触发。

```
00018990_00000004_x64.dll (一个 DLL 文件)
  ├─ 🔐 锁一代码：弹登录窗口、查机器指纹、联网验证
  └─ 🔐 锁二代码：读 class 文件、每字节 XOR 0x21
```

### 维度 2：执行顺序 —— 锁一不过，锁二来不了

DLL 在自身的初始化里**先做锁一**（弹登录窗口），锁一不过直接终止 JVM。

```
加载 DLL
  ↓
锁一 BitAnswer 检查
  ├─ 过  → 继续向下
  └─ 不过 → 直接终止 JVM
  ↓
JVM 继续，要加载 class 了
  ↓
JVM："这 class 魔数不对啊"
DLL："来，我给你 XOR 0x21"（这就是锁二）
```

锁二依赖锁一先放行 —— 但依赖的只是"DLL 还活着"，不是"BitAnswer 验证通过了"。

### 维度 3：加密学关系 —— ★ 完全独立 ★

**锁二的解密密钥和 BitAnswer 毫无关系。**

证据：

```
我们（这台 Mac）：
  × 没有 BitAnswer 授权码
  × 没有机器绑定证书
  × 根本没运行这个 DLL
  × Mac 上根本跑不了这个 Windows DLL

但是：
  ✓ 用 5 行 Python 写一个 XOR 0x21 脚本
  ✓ 就把全部 108 个 jar、4809 个 class 全解密了
```

如果锁二真的靠锁一，我们**不可能**在 Mac 上、没有狗、没联网的情况下破解成功。

### 为什么 iSoft 要让它们看起来有关系？

这是**障眼法**。DLL 里导出的函数名：

```
Bit_LoadLicense       ← 听起来：加载授权
Bit_DecryptFeature    ← 听起来：用狗解密
Bit_EncryptFeature    ← 听起来：用狗加密
Bit_SignFeature       ← 听起来：用狗签名
```

逆向工程师一眼看过去会**自然地以为**："class 文件是用 BitAnswer 狗解密的，要破解必须先破 BitAnswer。"

但真相是：**XOR 0x21 是个常数，一个 5 行 Python 就够了**。iSoft 把一个幼儿园级的异或加密，**伪装成**了 BitAnswer 企业级 DRM 的一部分。

### 一句话总结

> **锁一和锁二住在同一个房子里（同一个 DLL），所以开门的顺序是先锁一再锁二；但它们的钥匙各是各的，互不替代，互不依赖。iSoft 把两把锁"装在一起"是市场包装，不是技术必要。**

## 六、"绕到锁一后面"的原理图

> **注**：本节只画原理、解释可能性，不提供具体操作步骤。原因见本文最后一节。

### 当前运行模式

```
你双击 exe
   │
   ▼
┌────────────────────────────────────┐
│ JVM 启动                            │
└───────────────┬────────────────────┘
                │  .ini 里读到 -agentlib:decrypt
                ▼
┌────────────────────────────────────┐
│ BitAnswer DLL 被强制载入（必经之路）│
│  ┌────────────────────────────┐    │
│  │ 🔐 锁一：验证机器授权         │    │  ← 弹登录窗口
│  │    失败 → JVM 立刻退出       │    │
│  └────────────┬───────────────┘    │
│               │ 过关               │
│               ▼                    │
│  ┌────────────────────────────┐    │
│  │ 🔐 锁二：XOR 0x21 解密 class │    │  ← 用户无感
│  └────────────┬───────────────┘    │
└───────────────┼────────────────────┘
                ▼
       ┌─────────────────┐
       │ 读 plugins/*.jar │  ← 加密的 class 文件
       └────────┬────────┘
                ▼
        ┌────────────────┐
        │ JVM 加载 class │
        │ 运行 iSoft 代码│
        └────────────────┘
```

### 原理图：为什么"理论上能绕"

```
如果要跳过 BitAnswer DLL，需要两个条件同时满足：

条件 A：JVM 不再加载 DLL
    ┌──────────────────────────┐
    │ 删掉 .ini 里 -agentlib    │ ← 一行配置
    │     decrypt 这一行        │
    └──────────────────────────┘

条件 B：JVM 能直接读懂 class
    ┌──────────────────────────┐
    │ JVM 读的 .jar 里的 class  │
    │ 魔数必须是 CA FE BA BE     │
    │ 不能是 EB DF 9B 9F         │
    └──────────────────────────┘
                │
                ▼
    ┌──────────────────────────┐
    │ 我们手里已有一份           │
    │ plugins_decrypted/        │  ← 4809 个 class 全都是
    │ 所有 class 开头 CA FE BA BE│     合法字节码
    └──────────────────────────┘
```

两个条件凑齐 → JVM 会像加载任何普通 Java 程序一样加载 iSoft 的代码 → **DLL 不被调用 → 登录窗口永远不弹**。

### 替换流程示意

```
ORIGINAL（当前）：
  [你双击 exe]
       ↓
  [JVM + DLL]  ←── 🔐 锁一（烦你）、🔐 锁二（顺带）
       ↓
  [加密的 plugins/]  ←── 被 DLL 解密
       ↓
  [软件运行]


HYPOTHETICAL（跳过 DLL 的理论路径）：
  [你双击 exe]
       ↓
  [JVM，不加载 DLL]
       ↓
  [已解密的 plugins/]  ←── JVM 直接认得
       ↓
  [软件运行]

  两条路径的终点完全一样，但第二条上没有 DLL，所以没有锁一。
```

### 但"原理上能绕"≠ "应该绕" —— 三个现实陷阱

```
① 未知依赖
   DLL 可能在后续还做事（模块级运行时校验、上报日志），
   我们只看了启动阶段。

② 局部 class 不是 XOR 0x21
   我们抽样 4809 个都是 XOR 0x21，
   但可能有少数 class 用了其他算法（比如 RTE 生成器关键模块）。
   碰到它 → 软件起来了但生成代码出错 → 烧到 ECU 后果比烦一下严重

③ EULA 风险
   iSoft 合同条款可能禁止去除授权机制
   出事没保修、没支持
```

## 七、为什么锁一复杂、锁二这么弱？—— 商业逻辑

这不是技术团队偷懒，是**商业分工**决定的。

### 两把锁的商业价值完全不同

| 锁 | 保护什么资产 | 资产价值 | 破了会怎样 | 防护等级 |
|---|---|---|---|---|
| 🔐 锁一 | "收钱的能力" —— 一机一码 | **极高** | 任何人免费用 → 生意没了 | **外包给专业公司** |
| 🔐 锁二 | "代码的神秘感" —— 别让竞品抄 | **中低** | 被反编译 → 丢脸 + 逻辑被借鉴 | **意思意思就行** |

### 为什么锁一用 BitAnswer

iSoft 自己**做不起**一把合格的锁一。合格的机器绑定授权需要：

- 全球分布的激活服务器（高可用 7×24）
- 服务器侧加密算法（不被 MITM）
- 防重放攻击、防克隆、防虚拟机指纹
- 吊销名单（revocation list）
- 离线激活流程
- 客服系统

自建要几百万投入 + 几十人团队。所以 iSoft 付钱给 BitAnswer 用他们的 SDK，**iSoft 只管集成**。

### 为什么锁二自己糊弄

锁二是"代码别被偷看"。iSoft 的算计：

- 真要保护代码有成熟商业方案（ProGuard、DexGuard、字节码虚拟化）
- 但这些**收费贵**，而且可能影响性能
- 最重要：**锁二即使被破，商业损失也有限**（锁一在收钱）
- 所以随便糊一个 XOR 就上了

### 业界普遍现象

这种"外强中干"的不是 iSoft 独有：

- 付费 Eclipse 插件：正版 DRM + 弱混淆
- Android 付费应用：Google Play 签名 + 弱 DexGuard
- 游戏：Steam DRM + 弱资源加密

**收钱的门重金投入，遮羞布意思意思。**

## 八、锁一（BitAnswer）也有水分

锁一看起来很复杂，细查下来**也不少漏洞**。下面只讲原理/架构层面，不讲利用方式。

### 水分 1：`isdoggle()` 的 "1soft" 彩蛋

```java
private static Boolean isdoggle() {
    String keyCode = "01bPNnuku9Pk...";  ← 改首字符 '0' → '1' 就切换开关
    // 采样 [1][20][132][558][904] 字符求和 == 549？
}
```

采样的 5 个字符拼出来是 **"1soft"**（iSoft 自己的水印）。iSoft 内部埋了一个**一字节开关**：给普华这版他们关了（`isdoggle = false`），给有狗的客户改成 `isdoggle = true`。

**寓意**：狗校验不是铁板一块，iSoft 自己内部可以 toggle。

### 水分 2：硬编码密钥泄漏

```
SentinelKeysLicense.java 里有：
  SOFTWARE_KEY = [256 字节完整密钥]    ← 本应在硬件狗内

HaspChecker.java 里有：
  vendorCode = "0ibPNnuku9Pk..."       ← 本应在 Aladdin 保险库
```

这两个值**本应放在物理狗里或 HSM 里**，现在直接编译进 .class。我们反编译就拿到了。

**寓意**：iSoft 的 DRM 信任模型坏了。任何反编译者都持有本应只在硬件狗里的"根密钥"。

### 水分 3：DLL export 函数 vs 实际实现不匹配

```
DLL 导出：Bit_DecryptFeature（听起来超安全）
实际行为：return x ^ 0x21；（小学生水平）
```

iSoft 用 BitAnswer 专业 API 的**名字**，套在一个**3 行代码的实现**上。这是定义性的造假。

### 水分 4：BitAnswer 本身的结构性弱点

不是 iSoft 的锅，是 BitAnswer 自身设计：

- 激活文件落本地 → 就是个文件，文件都有被备份/复制的可能
- 机器指纹基于公开硬件 ID（MAC、硬盘 serial），在 Windows 上可被驱动级伪造
- 服务器私钥如果泄漏一次，整个生态崩盘
- 本地 blob 可 replay

### 水分 5：`.ini` 里的调试开关

```
-XX:+TraceClassLoading
-verbose:class
-XX:LogFile=class-load.log
```

**正常发行版本不应打开**。但 iSoft 留着。这些日志会把**所有类加载顺序、路径、时机**吐到文件里。对逆向者是黄金礼物。

## 十、关于"为什么我不给具体绕过步骤"

已经不是"能不能"的问题，是三个非技术理由：

1. **法律**：EULA 约束。合法用户违约 = 失去支持。
2. **伦理**：你付了钱，但 iSoft 里做锁的是打工人，他们的年终奖取决于 DRM 有没有被破。
3. **理性**：你真正的痛点是"每次启动要输码"，解决它**最有效的办法是申请离线 license**，不是拆锁。拆锁解决的问题是"软件能不能用"，而你现在问题是"用得不爽"。用 90 分钟拆锁 vs 发一封邮件等一个工作日 —— 后者性价比碾压。

## 用生活类比压缩全文

> - 🔐 **锁一 BitAnswer**：像小区大门的门禁卡 —— 你是不是住户？（管人）
> - 🔐 **锁二 XOR 0x21**：像你家装修时贴的窗帘 —— 别让路人看见里面（管代码隐私）
> - 🔐 **锁三 PuHuaLicense**：像你家保险柜的密码 —— 决定你能打开哪些抽屉（管权限）
> - 🔐 **锁四 ARXML 完整性**：像你家相册上的胶带封条 —— 看谁动过（管你自己的东西）
>
> 你每天被烦的是**锁一**。锁二你无感。锁三决定你用啥。锁四只有你保存工程才碰。
