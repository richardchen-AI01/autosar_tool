| Q | 选择 |
|---|---|
| **Q1** 数据 + 持久化 | **全 EMF** (read + save 都走 EMF) |

## 范围 (含 / 不含)

### 含
- 数据模型层全切 EMF EObject (从 Sphinx ResourceSet load 拿 typed model)
- 持久化全切 EMF (`Resource.save()`)
- Editor 切 Sphinx `AbstractFormPage` + EMF databinding (schema-driven，不再 hand-code 字段)
- ContentProvider 切 Sphinx `BasicExplorerContentProvider` (EMF 模型遍历)
- Generator + Validator 切 Xtend (in-process，吃 EObject)
- Algorithm 测试重定义：byte-equal → 语义等价
- 加 NvM 模块作为多模块验证 (单独 OSGi bundle)
- 引齐缺的 Sphinx UI / EMF databinding / nature bundle
- BSW workspace 自动加 `autosarnature + bswbuildernature` project nature

### 不含 (留 v0.3+)
- 其它 57 个模块（Det/Ea/Fee/Can/Com/Dem/Dcm 等等）
- iSoft 的 importers (DBC / FIBEX / ODX 等)
- def.editor (schema 编辑器)
- parallel model manager
- 商业级 license / installer

## 总估算

**18-25 天**（含每阶段撞坑迭代 + GUI 验等用户回应）。比上次估的 9-10 天多——加了 NvM 模块 + Xtend 翻译 + nature 等。

## 风险

| 风险 | 概率 | 缓解 |
|---|---|---|
| Sphinx 0.11.2 ResourceSet 加载 .arxml 在 Java 8 + Eclipse 2018-12 上撞 ClassCast / Xerces 内部 hack 又复发 | 中 | E1 spike 第一时间验，挂了立刻撞回 |
| EMF databinding 配 EcucContainerDef schema 自动渲染失败（schema metadata 不齐）| 中 | E4 早期 spike |
| Xtend 翻译 jinja 时一些动态特性翻不过去（Python 用 dict comprehension 等）| 高 | 翻译时遇到现搞，必要时落到 Java 实现 |
| 语义等价测试定义不严密，漏掉 regression | 高 | E5 写测试时仔细 review，每个字段 / 引用都覆盖 |
| NvM schema 比 MemIf 复杂得多 (Block descriptor 嵌套)，spike 才能算准 | 中 | E8 第一天起做 spike，挂了重新估 |
| Cross-module ref (MemIf ↔ NvM) 在 EMF model 中怎么 wire 不熟 | 中 | E8 后期撞坑 |

## 不切 v0.2 的代价

如果停在 v0.1 file-system 路线：
- 加每个新模块 ~500 行 hand-code (4 模块就 2000+ 行)
- Refactor / Search / Compare / Cross-ref 全部不能做
- v0.3 商业 release 必须切 EMF，那时 backlog 更大
- 实际**不切更贵**（已经在 sweep 报告里算过）
